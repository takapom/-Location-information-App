# 移動距離連動型陣取り位置情報アプリ「TERRI」 最終要件定義書

> **版数**: v1.1　**最終更新**: 2026-04-25　**方針**: Z世代UX最優先

---

## 1. プロダクト概要

本アプリケーション「TERRI（テリー）」は、ユーザーの移動距離に基づいて現実世界に仮想的な陣地（ポリゴン）を生成し、その陣地の面積を友達同士で競い合う位置情報アプリです。

「歩いた分だけ、世界が自分のものになる」というコンセプトのもと、Z世代の男女をメインターゲットとし、ZenlyやWhooのような「マップが常に生きている」ポップで直感的なユーザー体験（UX）を提供します。

---

## 2. ターゲットユーザーとUX設計原則

### 2.1. ターゲットユーザー
*   **Z世代（10代後半〜20代前半）の男女**
*   日常的に位置情報共有アプリ（Zenly, Whooなど）を利用している層
*   散歩、通学、通勤などの日常的な移動に「ちょっとしたゲーム性」を求めている層

### 2.2. Z世代UX設計の7原則
本アプリのUI/UXは、以下の7原則に基づいて設計されています。

1.  **即時フィードバック**: ボタンやアクションに対して「反応した感」を視覚的に強調する。
2.  **SNSシェア導線の最前面化**: 完了画面にInstagram Stories風のシェアカードを配置し、バイラル拡散を促す。
3.  **自己表現・カスタマイズ感**: プロフィールに絵文字ステータスを設定でき、陣地の色を自由に選択できる。
4.  **ゲーミフィケーションの可視化**: ランキングに「先週比+0.3km²」などの成長バッジを追加し、達成感を数値で強調する。
5.  **ゼロ摩擦のオンボーディング**: ログイン画面はSNS認証ボタンを最大化し、「1タップで始める」体験を提供する。
6.  **友達の存在感を常に感じさせる**: メインマップ上の友達アバターを大きく表示し、「今アクティブ」バッジでリアルタイム性を演出する。
7.  **余白と大きなタイポグラフィ**: Zenly的な「大きな地名テキスト」を活用し、情報密度を下げて直感的にする。

## 3. システムアーキテクチャ

本システムは、Cloudflareのエッジコンピューティング環境を中心とした分散型アーキテクチャを採用し、低遅延かつスケーラブルなサービス提供を実現します。

| 層 | 技術要素 | 役割 |
|---|---|---|
| **クライアント層** | React Native (Expo) | UI提供、GPS取得、地図描画（Mapbox Lightスタイル推奨） |
| **エッジロジック層** | Cloudflare Workers / Agents SDK (Durable Objects) | API受付、ユーザーごとの状態管理、ビジネスロジック実行 |
| **データストア層** | Cloudflare D1 (SQLite) | ユーザーごとの独立したデータ永続化（移動履歴、ポリゴンデータ） |

## 4. コア機能仕様（Phase 1 決定事項）

### 4.1. ポリゴン生成アルゴリズム
**バッファリング（軌跡の太線化）＋コンケーブハル（Alpha Shape）のハイブリッド方式**を採用します。

*   **バッファ半径**: 固定値 30m（徒歩・ランニングの平均的な道幅を考慮）。
*   **コンケーブハルの適用条件**: GPS点が20点以上、かつ軌跡が閉じている（出発点から500m以内に戻った）場合に内側の面積を追加算入。
*   **面積の重複カウント**: 他ユーザーの陣地との重なりは許容し、重なった部分も各ユーザーの面積としてカウントする。

### 4.2. GPSデータの送信タイミング
**リアルタイム逐次送信（ストリーミング方式）**を採用し、没入感を高めます。

*   **通常時の送信間隔**: 5秒ごと、または10m以上の移動を検知したとき。
*   **低速・停止時の送信間隔**: 移動速度が0.5m/s未満の状態が30秒継続した場合、送信間隔を30秒ごとに切り替えてバッテリーを節約する。

### 4.3. 陣地の確定条件
**「移動中のリアルタイムプレビュー表示」＋「アクティビティ終了時に自動確定」**の組み合わせを採用します。

*   **リアルタイムプレビュー**: トラッキング中、クライアント側でGPSデータをもとに半透明の暫定ポリゴンをリアルタイム描画する。
*   **陣地の確定処理**: ユーザーが「STOP」ボタンを押して終了したタイミングで、サーバー側が最終的なポリゴンを生成・面積を算出して確定する。

## 6. 友達の現在地リアルタイム表示

本機能は、友達関係にあるユーザーの現在地をメインマップ上にリアルタイムで表示するものです。ZenlyやWhooと同様に「友達が今どこにいるか」を常に感じられる体験を提供し、アプリへの継続的なエンゲージメントを生み出します。

### 6.1. 機能概要

| 項目 | 仕様 |
|---|---|
| **表示対象** | 友達関係にあるユーザーのうち、位置情報の共有を許可しているユーザーのみ |
| **表示要素** | 友達のアバター画像を現在地マーカーとして地図上に表示（Whoo風の円形アバター＋カラーリング） |
| **更新頻度** | 友達の位置情報は **15秒ごと** に自動更新する |
| **アクティブ表示** | 友達がトラッキング中の場合、アバターマーカーに「🔥 アクティブ」バッジを表示する |
| **最終更新時刻** | アバターマーカーをタップすると「〇分前に更新」の最終更新時刻を表示する |
| **オフライン時** | 最後に取得した位置情報を表示し、30分以上更新がない場合はマーカーをグレーアウトする |

### 6.2. プライバシー制御

位置情報の共有はユーザーが明示的に制御できる必要があります。

| 設定項目 | 詳細 |
|---|---|
| **共有のON/OFF** | プロフィール・設定画面（S11）から、自分の位置情報の共有を全体的にON/OFFできる |
| **友達ごとの制御** | Phase 2以降の拡張として、友達ごとに共有可否を設定できる機能を追加する |
| **共有停止時の表示** | 自分の位置情報共有をOFFにした場合、友達のマップ上から自分のマーカーが消える |
| **初回同意** | アプリ初回起動時のオンボーディングで、位置情報共有に関する明示的な同意を取得する |

### 6.3. 技術実装方針

友達の位置情報をリアルタイムに配信するため、以下の仕組みを採用します。

| 項目 | 仕様 |
|---|---|
| **配信方式** | Cloudflare Workers の **Durable Objects** を活用したサーバー送信イベント（SSE）またはポーリング（15秒間隔）を採用する |
| **データフロー** | 友達のUser Agent (Durable Object) が最新の位置情報を保持し、閲覧側のクライアントからのリクエストに応じて返却する |
| **位置情報の保存** | 友達の現在地はD1には保存せず、Durable Objectsのメモリ上にのみ保持する（プライバシー保護のため、過去の位置履歴は自分自身のデータのみ保存） |
| **送信データ形式** | `{ user_id, latitude, longitude, timestamp, is_active, avatar_url }` |

### 6.4. S04（メインマップ）への反映

友達の現在地表示はメインマップ（S04）の地図レイヤーに統合されます。

| UI要素 | 詳細 |
|---|---|
| **友達アバターマーカー** | 友達のプロフィール画像を円形にクリッピングし、友達ごとのカラーリング（コーラル/ミント/ラベンダーなど）のリングを付けて地図上に表示 |
| **アクティブバッジ** | トラッキング中の友達のマーカーには🔥バッジを重ねて表示し、動的な存在感を演出 |
| **「〇人が今アクティブ」バッジ** | マップ上部中央に「3人が今アクティブ🔥」のようなピル型バッジを常時表示し、友達の活動を促す |
| **マーカータップ時** | 友達のアバターをタップすると、名前・現在地の地名・最終更新時刻・総獲得面積をポップアップで表示 |

## 8. セキュリティとプライバシー

*   **通信の暗号化**: クライアントとCloudflare Workers間の全通信はHTTPS（TLS 1.3）で暗号化。
*   **認証・認可**: JWTベースの認証トークンを使用。他ユーザーのデータへのアクセスは、友達関係が確立されている場合のみ許可。
*   **ユーザーごとのデータ分離**: Cloudflare D1によりユーザーごとに独立したデータストアを構築し、不正アクセスリスクを最小化。
*   **GPS精度フィルタリング**: 精度（accuracy）が50m以上のGPSデータはノイズとして記録するが、ポリゴン生成の計算対象からは除外する。

## 9. 今後の検討事項（バックログ）

1.  **バッテリー消費の最適化**: バックグラウンドトラッキングによる消費を抑えるため、移動速度に応じたGPS取得頻度の動的制御を継続的にチューニングする。
2.  **GPS偽装（スプーフィング）対策**: 移動速度の異常値検知（例：時速200km以上の移動を検知した場合は無効化）をサーバー側で実装する。
3.  **ポリゴンの簡略化**: 長時間の移動では頂点数が膨大になるため、Douglas-Peuckerアルゴリズム等でポリゴンを間引き、描画パフォーマンスを維持する。
