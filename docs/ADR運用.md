# ADR運用

> 目的: TERRIのアーキテクチャ判断を、人間とCodexの両方が探索・理解・更新しやすい形で残す。  
> 対象: 技術選定、責務分割、データ設計、権限制御、位置情報処理、UI構造、テスト方針など、後続実装に影響する決定。

---

## 1. 基本方針

ADRは、TERRIにおけるアーキテクチャ上の判断を記録するための文書である。

ADRでは、単に「何を実装したか」ではなく、次の内容を後から再利用できる粒度で記録する。

- 何を決定したか
- なぜその決定に至ったか
- どの選択肢を検討したか
- なぜ他の選択肢を採用しなかったか
- どのレイヤー、ファイル、テーブル、policy、テストに影響するか
- 将来変更する場合、どの判断を見直すべきか

TERRIでは、`docs/architecture.md` を現時点の統合された設計判断として扱う。ADRは、その判断を補強、更新、具体化するために使う。

要件の根拠が必要な場合は、次の要件定義書を参照する。

```text
docs/移動距離連動型陣取り位置情報アプリ「TERRI」 最終要件定義書.md
```

TERRIにおける判断の関係は次のとおりである。

```text
要件定義
  ↓ 何を満たすべきか
docs/architecture.md
  ↓ 現時点の統合された全体設計
ADR
  ↓ 個別判断の背景・選択肢・決定・影響
実装 / テスト / レビュー
```

ADRは実装説明書ではなく、将来の判断材料である。  
実装詳細を書く場合も、決定の理解、影響範囲の特定、レビュー、テストに必要な範囲へ限定する。

---

## 2. ADRを書く基準

ADRは、すべての変更に対して作成するものではない。

ADRを書く対象は、後続の実装、設計、テスト、運用、レビューに影響する判断である。

ADRを書くべき判断は次のとおりである。

- アーキテクチャの構造を変える判断
- レイヤーや責務の分割に関する判断
- repository interface、service、hook、moduleなどの境界に関する判断
- DBテーブル、インデックス、RLS policy、権限制御に関する判断
- 位置情報、移動履歴、友達現在地、陣地計算など、プライバシーや正確性に関わる判断
- Supabase、PostGIS、Edge Functions、Realtimeなど、主要技術の採用・非採用に関する判断
- テスト方針、mock方針、contract test方針に関する判断
- 既存の `docs/architecture.md` を更新する必要がある判断
- 将来、なぜそうしたのかを再確認する可能性が高い判断

ADRを書かなくてよい変更は次のとおりである。

- 表記ゆれの修正
- 小さなバグ修正
- 既存方針の範囲内で完結する実装
- 外部仕様や責務分割に影響しない局所的なリファクタリング
- テストケースの単純な追加
- 既存ADRや `docs/architecture.md` にすでに根拠が明記されている変更

判断に迷う場合は、次の条件を満たすならADRを書く。

```text
その判断を将来変更するとき、
「なぜ当時この設計にしたのか」を知る必要があるか。
```

必要があるならADRを書く。  
必要がないなら、PR説明や実装コメントで十分とする。

## 3. architecture.md と ADR の関係

`docs/architecture.md` は、TERRIの現時点における統合された設計図である。

ADRは、個別の設計判断の理由、比較対象、影響範囲を残す判断ログである。

通常は、次の優先順位で参照する。

```text
1. docs/architecture.md
   ↓ 現時点の統合設計を確認する

2. 関連ADR
   ↓ 個別判断の理由、選択肢、影響範囲を確認する

3. 要件定義書
   ↓ 判断の根拠となる要件を確認する
```

ただし、承認済みADRが `docs/architecture.md` の内容を更新する決定である場合、そのADRには必ず次を記載する。

- `docs/architecture.md` のどの箇所を更新するのか
- なぜ既存設計を変更するのか
- 実装・レビュー時にどちらを優先するのか

承認済みADRと `docs/architecture.md` が矛盾する場合は、ADRに明示された更新対象と置き換え理由を確認する。

ADRに更新対象と置き換え理由が明記されている場合、その範囲ではADRの決定を優先し、`docs/architecture.md` を更新対象とする。

ADRに更新対象や置き換え理由が明記されていない場合は、`docs/architecture.md` を優先し、ADR側を要確認とする。

```text
docs/architecture.md
  ↓ 現在の統合設計

ADR
  ↓ 個別判断の理由・選択肢・影響範囲

承認済みADRが既存設計を更新する場合
  ↓
ADRに更新対象と置き換え理由を書く
  ↓
docs/architecture.md を更新する
```

## 4. 実装前方針と実装後ADR

TERRIのADRは、実装後にだけ書くものではない。

実装前の設計方針と、実装後に確定した判断を、同じ判断履歴として扱う。

基本は、1つのADRを次の流れで更新する。

```text
実装前
  ↓
ADRを「提案済み」で作成する
  ↓ 方針、選択肢、影響範囲、テスト観点を明確にする

実装 / テスト / レビュー
  ↓ 実装結果が方針と一致するか確認する

実装後
  ↓
同じADRを「承認」に変更する
  ↓ 実装結果、差分、関連ファイルを備考に追記する
```

実装前のADRは、Codexが実装に入る前の作業方針として使える粒度にする。

特に、次の情報を先に書く。

- 変更予定のレイヤー
- 変更予定のfeature
- repository interface
- DBテーブル
- RLS policy
- Edge Functions
- Supabase Realtime
- テスト観点
- mockと実実装の境界

実装後のADRは、実装で実際に採用された構造を正として更新する。

実装前の見込みと実装結果に差分がある場合は、次のように扱う。

- 決定の本質が変わらない場合
  - 同じADRを更新する
  - `決定` または `影響` に実装結果を反映する
  - `備考` に実装時の差分を残す

- 決定の本質が変わる場合
  - 既存ADRを無理に書き換えない
  - 実装前ADRを `棄却` に変更する
  - 新しいADRを作成する
  - 置き換え関係を双方の `備考` に明記する

例:

```text
ADR-0003 提案済み: Map画面の状態管理をContextへ集約する
  ↓ 実装中に、Contextでは責務が広すぎると判明

ADR-0003 棄却
  ↓ 備考に棄却理由と代替ADRを記録

ADR-0004 承認: Map feature内のhookとrepositoryへ状態を分離する
```

承認済みADRを後から変更する場合は、過去の判断を消さない。

承認済みADRの方針が現在の設計として採用されなくなった場合は、原則として新しいADRを作成して置き換える。既存ADRは当時の承認済み判断として残し、備考に置き換え関係を追記する。

```text
ADR-0005 承認: 友達現在地をRealtime Presenceで扱う
  ↓ 将来、保存要件が追加された

ADR-0012 承認: 友達現在地履歴を短期間保存する

ADR-0005 の備考:
- ADR-0012 により一部置き換え
```

この運用により、Codexは実装前には「何を目指すか」を読み、実装後には「実際に何が正となったか」をADRから判断できる。

## 5. 保存場所と命名

実装前方針と実装後ADRは、次のフォルダに分けて保存する。

```text
docs/
  adr-proposals/   # 実装前方針を書く
  adr/             # 実装後ADRを書く
```

| フォルダ | ファイル | 書く内容 |
|---|---|---|
| `docs/adr-proposals/` | `0001-<slug>.md` | 実装前の方針を書く。判断が必要になった背景、検討した選択肢、採用予定の方針、変更予定のレイヤー・ファイル・テーブル・policy、テスト観点、未決事項を書く |
| `docs/adr/` | `0001-<slug>.md` | 実装後に確定したADRを書く。実際に採用した決定、実装された変更範囲、実装前方針との差分、影響、確認したテスト、関連ファイル、置き換え関係を書く |
| `docs/architecture.md` | `architecture.md` | 現時点の統合された全体設計を書く。承認済みADRで全体設計が変わった場合は、ADRの決定に合わせて更新する |
| `docs/移動距離連動型陣取り位置情報アプリ「TERRI」 最終要件定義書.md` | 要件定義書 | 判断の根拠となる要件を確認する参照元として使う。ADRから参照するが、ADRのためだけに要件を書き換えない |

ファイル名は次の形式にする。

```text
docs/adr-proposals/0001-supabase-postgis-primary-datastore.md
docs/adr/0001-supabase-postgis-primary-datastore.md
```

同じ判断に対応する実装前方針と実装後ADRは、原則として同じ連番とスラッグを使う。

```text
実装前方針:
docs/adr-proposals/0001-supabase-postgis-primary-datastore.md

実装後ADR:
docs/adr/0001-supabase-postgis-primary-datastore.md
```

| 要素 | ルール |
|---|---|
| 連番 | 4桁のゼロ埋め。作成順に増やす |
| スラッグ | 英小文字、数字、ハイフンのみ |
| 拡張子 | `.md` |

タイトルは、決定の性質と文脈を十分に絞り込み、短く具体的にする。

良いタイトル:

```text
Supabase PostGISを主データストアに採用する
友達現在地履歴を永続化しない
画面からSupabaseを直接呼ばない
TrackingRepositoryでGPS点送信と活動終了を抽象化する
```

避けるタイトル:

```text
DBについて
位置情報の件
設計方針
対応内容
改善
リファクタ
```

タイトルは、次の条件を満たすようにする。

- 何に関する決定か分かる
- どの方向に決めたか分かる
- 対象範囲が広すぎない
- 後から検索しやすい語が含まれている

## 6. ステータス

ADRのステータスは次の3つだけを使う。

| ステータス | 意味 |
|---|---|
| 提案済み | 実装前方針。採用候補として記録し、実装・レビュー前の判断材料にする |
| 承認 | 採用された判断。実装・レビュー・テストはこの内容を前提にする |
| 棄却 | 採用しないと判断した案。代替ADRがある場合は備考で参照する |

ステータスは後から変更してよい。

ただし、過去の判断を消さず、変更理由を `備考` に残す。

許可する基本遷移は次のとおりである。

```text
提案済み → 承認
提案済み → 棄却
```

承認済みADRの内容を大きく変える場合は、既存ADRを書き換えすぎず、新しいADRを作成して置き換え関係を明記する。

```text
承認済みADR
  ↓ 方針を大きく変更したい
新しいADRを作成
  ↓
新ADRの備考に、置き換えるADRを記載
  ↓
旧ADRの備考に、置き換え先ADRを記載
```

`棄却` は、基本的に「提案されたが採用されなかった判断」に使う。

過去に採用されて実装された判断が、将来別の判断で置き換えられた場合は、原則として旧ADRを `棄却` に変更しない。旧ADRは当時の承認済み判断として残し、新しいADRで置き換え関係を示す。

## 7. 記載項目

ADRは必ず次の項目で構成する。

```text
# タイトル

## ステータス
## コンテキスト
## 検討した選択肢
## 決定
## 影響
## 備考
```

### 7.1 タイトル

タイトルは「何を、どの文脈で、どう決めたか」が分かる短文にする。

書き方:

- 決定対象を明示する
- 「採用する」「保存しない」「直接呼ばない」「分離する」「集約する」など、判断の方向を入れる
- 「設計」「改善」「対応」のような広すぎる語だけで終わらせない
- TERRI内で検索しやすいfeature名、table名、技術名、責務名を必要に応じて含める

例:

```text
# Supabase PostGISを主データストアに採用する
# 友達現在地履歴を永続化しない
# 画面からSupabaseを直接呼ばない
```

### 7.2 ステータス

ステータスは `提案済み`、`承認`、`棄却` のいずれかを1つだけ書く。

例:

```text
## ステータス

承認
```

### 7.3 コンテキスト

この決定を行った状況を書く。

コンテキストでは、単なる経緯ではなく、なぜ判断が必要になったのかを書く。

書き方:

- どの要件、制約、既存設計が判断の前提になったかを書く
- 何が問題なのかを書く
- 判断しない場合に何が困るのかを書く
- 比較対象や検討範囲を具体化する
- Codexが次回探索するときに必要なファイル、レイヤー、用語を明記する
- 必要に応じて、mermaidではないテキスト図で関係を可視化する

例:

```text
## コンテキスト

TERRIでは、GPS点、移動軌跡、確定済み陣地、ランキング集計を長期的に扱う。

これらは距離、面積、近傍検索、ポリゴン重なり判定を伴うため、通常のJSON保存だけでは計算の正確性と検索性能を担保しづらい。

判断対象:

Mobile
  ↓ GPS点送信
Backend
  ↓ 検証・確定処理
Database
  ↓ 空間検索・集計
Ranking / History
```

### 7.4 検討した選択肢

この決定に至るまでに検討した選択肢を書く。

ADRでは、採用した案だけでなく、採用しなかった案も記録する。

これにより、将来同じ議論を繰り返すことを防ぎ、Codexが「なぜ別案ではないのか」を理解できるようにする。

書き方:

- 比較した選択肢を列挙する
- 各選択肢の利点と欠点を書く
- 採用・不採用の理由を書く
- 不採用の選択肢も、判断に影響したものは残す

例:

```text
## 検討した選択肢

### 選択肢A: クライアント側で確定ポリゴンを生成する

利点:

- 即時表示と確定結果の差分が少ない
- Edge Function側の処理を減らせる

欠点:

- クライアントの計算結果を信頼する必要がある
- 不正なGPS点や改ざんへの耐性が弱い
- 面積計算や重なり判定の正確性を統一しづらい

採用可否:

不採用。

理由:

確定済み陣地はランキングや履歴に影響するため、クライアント側の結果を正としない。

### 選択肢B: Edge FunctionとPostGISで確定ポリゴンを生成する

利点:

- 確定処理をサーバー側に集約できる
- GPS点検証、面積計算、重なり判定を統一できる
- テスト対象を明確にできる

欠点:

- Edge FunctionとDB migrationの実装量が増える
- クライアント側のプレビュー結果と確定結果に差分が出る可能性がある

採用可否:

採用。

理由:

陣地確定処理の正確性、検証可能性、不正耐性を優先する。
```

### 7.5 決定

決定内容と根拠を書く。

書き方:

- 採用した選択肢を最初に明記する
- その選択肢を選んだ理由を、要件、制約、実装容易性、テスト容易性に接続して説明する
- 実装予定または実装済みの変更箇所を、ファイル、モジュール、関数、テーブル、policy、repositoryなどの単位で示す
- 変更箇所は、文章とテキスト図の両方で確認できるようにする
- `docs/architecture.md` を更新する必要がある場合は、更新対象を明記する

例:

```text
## 決定

確定済み陣地はPostgres/PostGISへ保存し、クライアント側のプレビュー結果を正としない。

クライアントはトラッキング中の即時表示だけを担当する。

STOP後の確定ポリゴン生成、GPS点検証、面積計算、重なり判定は、Edge FunctionとPostGISを経由して行う。

変更範囲:

apps/mobile/src/features/tracking
  ↓ プレビュー表示

supabase/functions/complete-activity
  ↓ GPS検証・確定処理

supabase/migrations
  ↓ activities / location_points / territories

packages/geo
  ↓ 共通ジオメトリ処理
```

### 7.6 影響

この決定で何が変わるかを書く。

影響では、良い影響だけでなく、受け入れる制約、移行コスト、テスト負荷も書く。

書き方:

- 影響するレイヤーを書く
- 影響するfeature、module、method、DB table、RLS policyを書く
- 追加・修正が必要なテストを書く
- 互換性、移行、パフォーマンス、プライバシーへの影響を必要に応じて分ける
- 影響がない領域も、誤解されやすい場合は明示する

例:

```text
## 影響

- Mobile:
  - `tracking` featureは暫定ポリゴンを表示する
  - 確定済みポリゴンは保存しない
  - STOP後はサーバー側の確定結果を取得する

- Edge Functions:
  - `complete-activity` がGPS点検証、ポリゴン生成、面積計算の境界になる

- Database:
  - `territories` は確定済みポリゴンだけを保持する
  - `location_points` は活動に紐づくGPS点を保持する
  - 必要に応じてPostGIS indexを追加する

- RLS:
  - `territories` のSELECT条件は、本人または友達関係が成立しているユーザーに限定する

- Test:
  - mock repositoryとSupabase repositoryの契約を揃える
  - GPS点の正常系、異常系、境界値をテストする
  - 不正なGPS点のnegative testを追加する

- Privacy:
  - 友達現在地のPresence情報はこの決定の保存対象に含めない
```

### 7.7 備考

この決定のメタデータを書く。

最低限、次を含める。

| 項目 | 内容 |
|---|---|
| 作成日 | ADRを作成した日付 |
| 更新日 | 最終更新日 |
| 関連ADR | 依存、置き換え、棄却理由に関係するADR |
| 置き換え関係 | このADRが置き換えるADR、またはこのADRを置き換えるADR |
| 関連ファイル | 判断に関係する主要ファイル |
| 参照資料 | 要件定義、architecture、外部資料など |
| 実装時の差分 | 実装前方針と実装結果に差分がある場合の記録 |
| 未決事項 | 今回のADRでは決めないが、将来判断が必要な事項 |

例:

```text
## 備考

- 作成日: 2026-04-26
- 更新日: 2026-04-26
- 関連ADR: なし
- 置き換え関係: なし
- 関連ファイル:
  - docs/architecture.md
  - packages/geo/src/territory-builder.ts
  - supabase/functions/complete-activity/index.ts
- 参照資料:
  - docs/architecture.md
  - docs/移動距離連動型陣取り位置情報アプリ「TERRI」 最終要件定義書.md
- 実装時の差分:
  - なし
- 未決事項:
  - なし
```

## 8. 人間とCodexが再利用しやすい書き方

ADRは、人間だけでなくCodexが実装、レビュー、修正に使えるように書く。

そのため、抽象的な方針だけで終わらせず、探索可能な具体名を含める。

| 観点 | 書き方 |
|---|---|
| 探索性 | 関連するディレクトリ、ファイル、feature名、table名、関数名を明記する |
| 責務境界 | UI、hook、repository、service、Edge Function、DBのどこに責務を置くかを書く |
| 差し替えやすさ | mockとSupabaseの境界、repository interface、contract testへの影響を書く |
| プライバシー | 位置情報、友達現在地、履歴保存、RLSへの影響を書く |
| テスト容易性 | どのレイヤーにどの成功・失敗・境界テストが必要かを書く |
| 将来変更 | 置き換え条件、未決事項、棄却した代替案を書く |

避ける表現:

```text
いい感じにする
適切に保存する
必要に応じて対応する
今後検討する
あとで直す
設計を改善する
```

推奨する表現:

```text
`friends presence` はRealtime Presenceで配信し、Postgresには保存しない。

`TrackingRepository` はGPS点の追加、活動終了、確定結果取得を同じ契約で提供する。

`territories` のSELECTは、本人または友達関係が成立しているユーザーに限定する。

`complete-activity` はGPS点検証、確定ポリゴン生成、面積計算を担当する。

Mobile側はトラッキング中の暫定表示だけを担当し、確定済み陣地の保存判断を行わない。
```

## 10. レビュー観点

ADRを作成・更新するときは、最低限次を確認する。

- タイトルだけで決定の対象と方向が分かる
- ステータスが `提案済み`、`承認`、`棄却` のいずれかである
- コンテキストに、判断前の制約、問題、比較対象が書かれている
- 検討した選択肢に、採用案と不採用案が書かれている
- 決定に、採用内容と根拠が書かれている
- 決定に、実装予定または実装済みの変更範囲が書かれている
- 影響に、具体的なレイヤー、ファイル、テーブル、メソッド、RLS policy、テスト観点が書かれている
- 備考に、作成日、更新日、関連ADR、置き換え関係、関連ファイル、参照資料がある
- 実装前ADRの場合、実装後に確認すべき差分観点が書かれている
- 実装後ADRの場合、実装結果が決定と影響に反映されている
- mermaidを使わず、必要な図は `text` コードブロックで表現している
- `docs/architecture.md` と矛盾する場合は、ADR側に置き換え理由と更新対象が書かれている
- 過去の判断を消さず、変更理由が備考に残っている

## 11. ADR更新時のルール

ADRを更新するときは、次のルールを守る。

### 11.1 提案済みADRを承認する場合

実装、レビュー、合意の結果、方針が採用された場合は、ステータスを `承認` に変更する。

その際、次を確認する。

- 実装結果が `決定` に反映されている
- 実際の影響範囲が `影響` に反映されている
- 関連ファイルが `備考` に記載されている
- 実装前方針との差分があれば `備考` に記載されている

### 11.2 提案済みADRを棄却する場合

提案済みの判断を採用しない場合は、ステータスを `棄却` に変更する。

その際、次を記載する。

- 棄却理由
- 採用しなかった選択肢
- 代替ADRがある場合はそのADR番号とタイトル
- 棄却によって影響を受ける実装・設計

### 11.3 承認済みADRを置き換える場合

承認済みADRの判断を将来変更する場合は、原則として新しいADRを作成する。

旧ADRは当時の判断として残し、新ADRで置き換え関係を明記する。

```text
旧ADR:
- ステータス: 承認
- 備考:
  - ADR-0012 により一部置き換え

新ADR:
- ステータス: 承認
- 備考:
  - ADR-0005 を一部置き換え
```

承認済みADRを直接大きく書き換えることは避ける。

ただし、誤字修正、関連ファイルの追記、実装時の差分追記など、判断の本質を変えない更新は同じADRに反映してよい。

## 12. 最小サンプル

```markdown
# 友達現在地履歴を永続化しない

## ステータス

承認

## コンテキスト

TERRIでは、友達の現在地を地図上に表示する。

友達現在地はリアルタイム性が重要であり、ランキング、陣地確定、履歴閲覧の正規データとしては使わない。

一方で、友達現在地をPostgresに保存すると、位置情報の保存範囲が広がり、プライバシーリスクとRLS設計の負荷が増える。

判断対象:

Mobile
  ↓ 友達現在地表示
Realtime Presence
  ↓ 一時的な位置情報配信
Postgres
  ↓ 保存しない

## 検討した選択肢

### 選択肢A: 友達現在地をPostgresに保存する

利点:

- 過去位置の参照が可能になる
- 障害時に再取得できる

欠点:

- プライバシーリスクが増える
- RLS policyが複雑になる
- 要件上、友達現在地履歴は必須ではない

採用可否:

不採用。

理由:

友達現在地は一時的な表示用途であり、履歴保存する必要がない。

### 選択肢B: 友達現在地をRealtime Presenceで扱い、Postgresには保存しない

利点:

- 保存対象を最小化できる
- プライバシーリスクを抑えられる
- 現在地表示の用途に合っている

欠点:

- 過去位置は参照できない
- Presence切断時の再取得はできない

採用可否:

採用。

理由:

友達現在地はリアルタイム表示のための一時情報であり、永続化しない方が要件とプライバシー制約に合う。

## 決定

友達現在地履歴はPostgresに永続化しない。

友達現在地はRealtime Presenceで配信し、Mobile側で地図表示に利用する。

Postgresには、友達現在地の履歴、最新位置、Presence状態を保存しない。

変更範囲:

apps/mobile/src/features/map
  ↓ 友達現在地を表示する

supabase/realtime
  ↓ Presenceで一時的な位置情報を配信する

supabase/migrations
  ↓ 友達現在地履歴用のtableは作成しない

## 影響

- Mobile:
  - 友達現在地はRealtime Presenceから取得する
  - アプリ再起動後に過去の友達位置は復元しない

- Database:
  - 友達現在地履歴用のtableは作成しない
  - 友達現在地用のRLS policyは不要

- Privacy:
  - 友達現在地の保存範囲を最小化できる
  - 履歴保存によるプライバシーリスクを避ける

- Test:
  - Presence受信時に地図上へ反映されることをテストする
  - Presence未接続時に友達現在地が表示されないことをテストする

## 備考

- 作成日: 2026-04-26
- 更新日: 2026-04-26
- 関連ADR:
  - なし
- 置き換え関係:
  - なし
- 関連ファイル:
  - docs/architecture.md
  - apps/mobile/src/features/map
- 参照資料:
  - docs/architecture.md
  - docs/移動距離連動型陣取り位置情報アプリ「TERRI」 最終要件定義書.md
- 実装時の差分:
  - なし
- 未決事項:
  - なし
```
